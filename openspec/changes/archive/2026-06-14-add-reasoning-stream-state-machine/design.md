# Design: 推理过程流式渲染的 P0/P1/P2 状态机

## Context
体验目标是「深度思考优先 + 打字机动态渲染 + 静态加载仅作最终兜底」。现状盘点：

- **后端已是真·流式**：`chatModel.Stream` 强制 `body["stream"]=true`，逐帧下发 `message`/`reasoning` 增量；仅当流式开启失败（setup 错误、供应商返回非 SSE 缓冲响应）时降级到 `fallbackStream`，把一次性结果**重新切片**为帧。即 P0 的后端侧已满足，降级路径也保持了分片节奏。
- **前端已有打字机**：`controller.ts` 的 `pump()` 以固定节拍逐字吐出 `assistant` 与 `reasoning`；`turn_start` 即显示 `LoadingBubble`（三点）。

真正缺口只有两点：
1. 等待态语义单一——三点气泡不区分「正常思考首包在途」与「本轮不会流式」。
2. 前端无法得知本轮是否走了降级路径，只能靠超时猜测兜底。

## Goals / Non-Goals
- Goals：把发送后的等待态收敛为 P0/P1/P2 三级；让 P2 可被后端信号确定性触发；P1 为毫秒级轻量默认态。
- Non-Goals：重写流式消费逻辑；改动思考块折叠或空轮兜底；引入「用户可关闭流式」开关（模型由服务端配置，流式默认恒为开）。

## Decisions

### D1：流式能力标记走 turn_start 的加法字段
`turn_start` 事件 `Data` 新增 `streaming bool`（缺省视为 `true`）。绝大多数轮在 `turn_start` 时即已知会走流式路径，故默认 `true` 让前端直接进入 P1。

**降级是迟到信息**：`fallbackStream` 的降级判定发生在模型调用阶段（晚于 `turn_start`）。因此不在 `turn_start` 回填，而是新增一个独立的轻量信号——复用既有 `turn_start` 事件类型再发一次、或更简单地在降级发生时发一个 `Data:{degraded:true}` 的 `turn_start`——让前端把当前轮切到 P2。

- 选定方案：**降级时补发一个 `turn_start{streaming:false, degraded:true}`**。理由：复用既有事件类型，旧客户端按「未知字段忽略 + 重复 turn_start 幂等」处理不受影响；前端只需在该事件里读 `streaming` 决定 P1/P2，无需新增事件类型。
- 替代方案（未采纳）：新增 `EventStreamMode` 事件类型。更显式但增加协议面，且与「加法式演进、未知类型忽略」相比并无收益。

### D2：P2 双保险触发（后端信号 + 前端超时）
P2 由以下任一条件触发：
1. 后端信号：收到 `turn_start.streaming === false`（含降级补发的 `degraded:true`）。
2. 前端超时：进入 P1 后约 **1500ms** 仍未收到任何 `message`/`reasoning` 增量。

两者是 OR 关系（双保险）：后端信号覆盖「已知非流式/已降级」，前端超时覆盖「网络彻底阻塞、信号也到不了」。任一首包增量到达即取消超时并进入 P0。

阈值 1500ms 的取舍：P1 要求「毫秒级」轻量呈现，1.5s 足够让正常首包（通常数百 ms 到 1s）停留在 P1 不闪烁，又不至于让真卡顿的轮长时间停在无明确加载语义的微提示上。阈值作为前端常量，便于后续调参。

### D3：P1 取代现有三点气泡为默认等待态
`LoadingBubble` 重构为可表达三级语义的等待组件（或拆为 P1 微提示 + P2 静态兜底两个组件）：

- P1：轻量微动效 + 「正在启动深度思考…」文案，符合「第一时间看到思考轨迹」的意图（即便思考增量尚未到，也先用文案提示思考正在启动）。
- P2：更明确的静态局部加载态（可沿用更醒目的 spinner/skeleton），仅在 D2 条件下出现。

状态机存活于单个「等待气泡」chat item，避免在 P1↔P2 之间插入/删除不同 item 造成布局跳动；切换仅改其内部呈现级别。首个增量到达时按现有逻辑 `clearLoading()` 移除并转入 P0 打字机。

### D4：超时管理与生命周期
- 超时计时器在 `turn_start`（streaming 默认 true 进入 P1）时启动；在首个 `message`/`reasoning` 增量、`turn_end`、`error`、`cancel` 时清除。
- 收到降级 `turn_start{streaming:false}` 时立即切 P2 并清除超时（信号已先于超时给出确定结论）。
- 与既有中断/队列逻辑兼容：中断（`cancel_turn`）与 `turn_end` 均须清理等待态计时器，防止悬挂计时器在下一轮误触发。

## Risks / Trade-offs
- **重复 turn_start 的幂等性**：前端 `turn_start` 处理需对「同一轮内第二次 turn_start（降级信号）」幂等——不重置 `producedRef`、不重复插入等待气泡，仅更新等待级别。已在 D1/D4 约束。
- **降级信号竞态**：降级信号可能与首个补发增量几乎同时到达。无论先后，结果一致（P2 或直接 P0 打字机），不会卡死；若增量先到则进入 P0，降级信号到来时已无等待气泡可切，按幂等忽略。
- **阈值误差**：1.5s 偏短可能在偶发慢首包时短暂闪现 P2。可接受——P2 仍是加载态而非错误，且双保险优先采用后端信号这一更准确来源。

## Migration / Rollout
纯加法式协议演进：`turn_start` 新增字段，旧前端忽略即退回「单一三点气泡」行为，无破坏。前后端可分别发布。
