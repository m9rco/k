# frontend-experience delta

## ADDED Requirements

### Requirement: 前端技术栈与构建
前端 SHALL 基于 Vite + React + TypeScript 实现，并以 Tailwind CSS 作为样式框架；构建产物 SHALL 经 Go 的 `embed` 打包进单可执行文件，保持「单二进制」部署形态不变。开发期 SHALL 通过 dev server 代理将 `/api`、WebSocket 与 SSE 转发到 Go 后端。后端 API/WS/SSE 契约 SHALL 保持不变。

#### Scenario: 单二进制仍可独立运行
- **WHEN** 执行生产构建（前端构建 → Go 构建）并运行生成的可执行文件
- **THEN** 无需额外静态资源服务即可打开完整前端
- **AND** 所有页面资源由该二进制内嵌提供

#### Scenario: 开发期实时通道可用
- **WHEN** 在开发模式下运行前端 dev server 与后端
- **THEN** 对话 WebSocket 与任务 SSE 经代理正常连通
- **AND** 前端能实时收到消息增量与任务进度

#### Scenario: 后端契约不变
- **WHEN** 前端重写完成
- **THEN** 后端的会话/实时/工作区/裁剪/下载/优化等接口形状未发生变化
- **AND** 仅前端的消费实现发生变化

### Requirement: 统一设计 token 与去 AI 化视觉
系统 SHALL 建立统一的设计 token 并在全局一致应用，达成沉静、专业、低装饰的视觉：圆角统一为 12px（小尺寸 8px，仅 chip/头像/状态点用全圆角）、正文行高 1.6、弱化边框（以轻分隔线与留白替代重边框）、移除装饰性渐变色块、强调色降饱和并克制使用。界面 SHALL 不再呈现典型「AI 模板」的多彩色块与重投影。

#### Scenario: 圆角与行高统一
- **WHEN** 渲染任意卡片、面板、输入框、按钮
- **THEN** 其圆角统一为 12px（或规定的 8px 小尺寸 / 全圆角特例）
- **AND** 正文文本行高为约 1.6，不显拥挤

#### Scenario: 去除装饰性色块
- **WHEN** 浏览主要界面区域
- **THEN** 不出现纯装饰性的渐变色块或高饱和填充
- **AND** 视觉以中性背景层级与留白区分结构，而非重边框或彩色块

#### Scenario: 设计 token 单一来源
- **WHEN** 调整圆角/行高/分隔线/强调色等基础视觉
- **THEN** 通过统一的 token 配置生效，而非散落的硬编码值

### Requirement: 组件库与可访问交互
复杂交互组件（弹窗、下拉/右键菜单、Tab、Toast、Tooltip 等）SHALL 使用 shadcn/ui（基于 Radix UI）组合实现，SHALL NOT 手写这些组件的开合/焦点/键盘可达逻辑。

#### Scenario: 弹窗与菜单基于组件库
- **WHEN** 打开图片预览弹窗、资产右键菜单或尺寸选择器的渠道 Tab
- **THEN** 这些交互由 shadcn/Radix 组件承载
- **AND** 支持 Esc 关闭、焦点管理与基本 aria 可达性

#### Scenario: 不手写复杂交互
- **WHEN** 新增或修改弹窗/下拉/Tab 类交互
- **THEN** 通过组合现有组件实现，而非手写其底层交互逻辑

### Requirement: 丝滑微动效
系统 SHALL 使用 Framer Motion 为关键交互提供克制的微动效（短时长、缓出、小位移、以透明度为主），并 SHALL 尊重用户的 `prefers-reduced-motion` 偏好（开启时弱化或关闭动效）。

#### Scenario: 关键交互有微动效
- **WHEN** 助手消息涌现、思考块展开/折叠、工具卡片状态切换、占位骨架替换为产物、卡片进出场
- **THEN** 这些过渡呈现平滑的微动效而非生硬跳变

#### Scenario: 尊重减弱动效偏好
- **WHEN** 用户系统开启「减弱动态效果」
- **THEN** 动效被弱化或关闭，交互仍可正常完成

### Requirement: 图标与品牌标识
系统 SHALL 使用 Lucide React 线性图标，SHALL NOT 使用彩色 Emoji 或 FontAwesome 表达功能/状态/品牌；品牌标识 SHALL 为简洁的单色线性 mark。

#### Scenario: 工具与操作使用线性图标
- **WHEN** 渲染工具调用卡片、菜单项、状态与操作按钮
- **THEN** 图标统一来自 Lucide 线性图标集
- **AND** 不出现 🎨/🎬/✂️ 等彩色 Emoji

#### Scenario: 品牌标识单色化
- **WHEN** 展示产品品牌标识
- **THEN** 使用简洁单色线性 mark，与整体克制风格一致

## MODIFIED Requirements

### Requirement: 品牌化科技感界面
系统 SHALL 提供精简、克制、专业的品牌化界面，传达明确的产品与服务意识，并在全站使用统一的视觉语言（配色、留白、图标、动效节奏）以建立成熟 Agent 产品的观感。视觉基调 SHALL 沉静、低装饰——避免典型「AI 生成模板」的多彩装饰色块与重投影，以中性层级与留白组织结构。系统 SHALL 通过统一的设计 token 组织视觉语言（圆角 12px、正文行高约 1.6、弱化边框、降饱和强调色），并在品牌区、按钮、卡片、弹层等关键元素上提供克制的微交互动效以增强质感；所有非必要动画 SHALL 在用户系统开启「减少动态效果」（prefers-reduced-motion）时降级关闭。

#### Scenario: 首屏品牌呈现
- **WHEN** 用户进入应用
- **THEN** 界面以精简、沉静的视觉呈现品牌与核心能力入口
- **AND** 顶部明确传达"宣发素材助手"的产品与服务定位

#### Scenario: 统一视觉语言
- **WHEN** 用户在对话区、工作区、弹层、通知之间切换
- **THEN** 各区域沿用一致的配色、12px 圆角、间距与过渡节奏，无割裂感

#### Scenario: 沉静低装饰基调
- **WHEN** 浏览任意主要界面
- **THEN** 不出现纯装饰性彩色块或重投影
- **AND** 结构以中性背景层级、留白与轻分隔线区分

## ADDED Requirements

### Requirement: 功能行为全量平移
前端重写 SHALL 完整保留现有全部能力的行为，仅改变视觉与交互质感，不得造成功能回归。涵盖：对话流式输出与思考打字机及折叠、工具调用卡片生命周期、工作区里程碑分组、长任务即时占位与演出式进度、失败任务的重试/移除/一键清除、多图参考改图入口、已生成图二次调整、视频产物预览、分层尺寸选择器（渠道→类型→尺寸）、上传、单图下载与打包下载、上下文状态面板与清理、提示词优化入口、偏好角落、下一步建议、工作区拖拽排序、响应式布局。

#### Scenario: 能力清单逐项可用
- **WHEN** 重写完成后逐项验收现有能力
- **THEN** 每项能力的用户可见行为与重写前一致或更好
- **AND** 不出现已有能力缺失或退化

#### Scenario: 实时体感保持
- **WHEN** 发起生图/二次调整等长任务
- **THEN** 工作区即时出现占位骨架并以演出式进度平滑推进，完成后回填产物
- **AND** 对话区的回答与思考仍以逐字方式实时呈现
