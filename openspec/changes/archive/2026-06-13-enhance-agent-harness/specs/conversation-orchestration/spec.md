# conversation-orchestration — Delta

## MODIFIED Requirements

### Requirement: Context 滑动窗口管理
系统 SHALL 对会话消息维护一个 token 预算受限的滑动窗口：超出预算时保留 system 提示与最近若干轮原文，对更早轮次做摘要压缩为单条 summary 消息，以防止 context 膨胀导致模型输出失真。token 估算 SHALL 对 CJK 字符与 ASCII 字节分别估算以降低中文场景的偏差。系统 SHALL 能在每轮结束时向前端下发当前窗口状态（估算 token、预算、是否已压缩）。

#### Scenario: 超预算触发压缩
- **WHEN** 会话消息累计超过配置的 token 预算
- **THEN** 系统将较早的历史轮次压缩为一条 summary 消息并保留最近轮次原文
- **AND** 后续模型调用使用压缩后的窗口

#### Scenario: 大块工具结果不入 context
- **WHEN** 工具返回大体积结果（如图片二进制/base64）
- **THEN** 系统仅以引用 id 形式将结果纳入对话上下文，不将原始大块数据送入 LLM context

#### Scenario: CJK 与 ASCII 分别估算 token
- **WHEN** 会话消息混合中文与英文/符号
- **THEN** 系统对 CJK 字符与 ASCII 字节使用不同的字符/ token 比率分别估算并合计
- **AND** 该估算用于窗口压缩决策

#### Scenario: 窗口状态下发
- **WHEN** 一轮对话结束
- **THEN** 系统向前端提供当前窗口的估算 token、预算与是否已压缩的状态

### Requirement: 流式对话输出
系统 SHALL 以**真·流式**方式消费模型供应商的流式响应，并将 agent 的回复增量与思考增量分别实时推送给前端，而非先获取完整结果再切块模拟。当模型返回思考内容（reasoning/thinking）时，系统 SHALL 将其作为独立于回答正文的增量类型推送；对于支持显式开启思考的会话供应商（如 Anthropic extended thinking），系统 SHALL 在请求中开启思考能力，使思考增量得以产生并下发，从而保证前端的思考块有内容可逐字渲染；当供应商流式解析失败时，系统 SHALL 降级为读取完整响应后补发，保证前端不空屏。此外，系统 SHALL 在收到用户消息后、调用模型之前立即发出一个"轮开始"信号，并在该轮结束（含错误或产出反问）时发出"轮结束"信号，使前端的 loading 态与模型首个增量解耦。

#### Scenario: 回答增量流式推送
- **WHEN** agent 生成回复
- **THEN** 系统按供应商返回的增量片段实时推送，前端可逐步渲染而非等待完整结果

#### Scenario: 思考增量独立推送
- **WHEN** 模型在回答前返回思考内容
- **THEN** 系统将思考增量以区别于回答正文的事件类型逐片推送
- **AND** 前端据此实时渲染思考块

#### Scenario: 开启供应商思考能力
- **WHEN** 当前会话供应商支持显式开启思考（如 Anthropic extended thinking）
- **THEN** 系统在模型请求中开启思考能力并设置合理的思考预算与输出上限
- **AND** 由此产生的思考增量经既有思考事件类型实时下发给前端

#### Scenario: 流式失败降级
- **WHEN** 供应商流式响应解析失败
- **THEN** 系统降级为读取完整响应并补发增量
- **AND** 前端最终仍能完整呈现回答，不出现空屏

#### Scenario: 轮开始信号即时下发
- **WHEN** 系统接收到用户消息并准备处理
- **THEN** 系统在调用模型之前立即发出"轮开始"信号
- **AND** 该信号早于任何思考或回答增量，使前端可即时进入 loading 态

#### Scenario: 轮结束信号收尾
- **WHEN** 一轮处理结束（正常完成、出错或产出反问 capsule）
- **THEN** 系统发出"轮结束"信号并携带本轮是否调用工具、是否产出 capsule 的元信息
- **AND** 前端据此结束 loading 态

## ADDED Requirements

### Requirement: 意图澄清与结构化反问
当用户请求命中能力白名单但**缺少安全调用工具所必需的信息**（例如未指明要操作的图、未说明改成什么、未给出目标尺寸/平台）时，系统 SHALL 通过澄清工具发起一次结构化反问，而不是猜测调用工具或仅以文字泛泛确认。反问 SHALL 包含一句问题与 2-4 个具体选项，每个选项 SHALL 同时支持"点击直接选择"与"作为可编辑文本由用户改写后提交"。系统 SHALL 能接住用户对反问的选择或改写输入，并据此续接同一会话的后续处理。

#### Scenario: 意图不明确时发起反问
- **WHEN** 用户请求命中能力但缺少必要信息（如"帮我改改这张图"未说明如何改）
- **THEN** 系统调用澄清工具返回一句反问与 2-4 个具体选项
- **AND** 本轮不调用生成/裁剪等执行类工具

#### Scenario: 用户点击选项续接
- **WHEN** 用户点击反问给出的某个选项
- **THEN** 系统以该选项对应的值作为后续输入续接处理
- **AND** 在信息已充分时调用对应工具完成任务

#### Scenario: 用户改写选项后提交
- **WHEN** 用户在选项的可编辑文本基础上改写并提交
- **THEN** 系统以改写后的文本作为后续输入续接处理

#### Scenario: 信息充分时不反问
- **WHEN** 用户请求已包含调用工具所需的全部必要信息
- **THEN** 系统直接调用对应工具，不发起多余反问

### Requirement: 面向 Web 的输出格式规范
系统 SHALL 约束 agent 的文本回复为面向 web 渲染的简洁纯文本，不输出 markdown 标记（标题井号、强调星号、表格、围栏代码块等）。当需要让用户在多个具体取值之间做选择时，系统 SHALL 通过澄清工具产出结构化选项，而非在文本里罗列编号选项。

#### Scenario: 文本回复不含 markdown 语法
- **WHEN** agent 生成面向用户的文本回复
- **THEN** 回复为纯文本，不包含 markdown 标记语法

#### Scenario: 选择类交互走结构化选项
- **WHEN** agent 需要用户在多个具体取值间选择
- **THEN** agent 通过澄清工具产出结构化选项而非文本编号列表

### Requirement: 分层 System Prompt
系统 SHALL 以分层结构组织 agent 的 system prompt，至少包含：角色定义、能力白名单、工具使用规范、交互与澄清规范、输出格式规范、安全规范、语言要求。其中 SHALL 包含"意图缺少必要信息时先澄清再行动"的指令，并保留既有的同轮工具调用、白名单拒绝、未配置告警、引用 id 使用、prompt 注入防护、简体中文回复等约束。

#### Scenario: prompt 含澄清优先指令
- **WHEN** 构建 system prompt
- **THEN** prompt 含"意图缺少必要信息时应先调用澄清工具反问而非猜测"的指令

#### Scenario: 既有约束保留
- **WHEN** 构建 system prompt
- **THEN** 同轮工具调用、白名单礼貌拒绝、未配置告警、引用 id 使用、prompt 注入防护、简体中文回复等既有约束均在对应层中保留
