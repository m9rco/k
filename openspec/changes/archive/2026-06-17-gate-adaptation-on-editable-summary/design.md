# Design

## Context

视觉分析（grok-4-fast）目前在 `adapt_to_platform` 工具内同步执行：`visionThemeReport` 发布参考图 → 调 grok 流式分析 → 把报告通过 `NotifyAnalysis` 推到前端折叠面板 → **立即 return**，工具拿报告作为 THEME 约束开始 AI 重绘。整个过程在**一次工具调用内**完成，没有 turn 边界，没有用户介入点。

本次要在「报告产出」和「AI 重绘」之间插入一个 3 秒可编辑确认窗口，且默认（不编辑）行为是无缝继续。核心难点：**在一次进行中的工具调用内部如何等待一个来自前端的人工信号**，同时不破坏既有「不修改就顺滑往下走」的体验。

## Goals / Non-Goals

Goals:
- 报告产出后、重绘前给用户 3 秒就地编辑摘要的机会
- 不编辑则零点击自动继续（倒计时到期默认提交 grok 原版）
- 编辑版回写缓存，同批图后续复用编辑版
- 加法式协议演进，旧客户端不受影响

Non-Goals:
- 不改造 grok 分析本身的 prompt/格式
- 不为 `edit_image`/生图/生视频等其他工具引入确认窗口（仅 `adapt_to_platform` 的分析阶段）
- 不做摘要的服务端格式校验/纠错（用户编辑内容按既有 `maxReportLen` 截断即可，不做结构强校验）
- 不持久化确认态跨进程重启（沿用 `vision_reports` 进程内/SQLite 既有生命周期约定）

## Decisions

### Decision 1：门控用「工具内阻塞等待 confirm 信号」，而非 capsule turn 边界

两个候选：

- **A. 复用 capsule 机制**：结束当前 turn，发 capsule「确认摘要？」，用户回传后开新 turn 续接适配。
- **B. 工具内阻塞**：`visionThemeReport` 报告产出后，在工具协程内阻塞等待一个 per-session 的 confirm channel，前端回传确认值（或倒计时到期默认值）后解除阻塞，**同一次工具调用**继续重绘。

选 **B**。理由：

- capsule（A）会把适配拆成两个 turn，模型需要在第二个 turn「记得」要继续适配——依赖模型时序，且 `adapt_to_platform` 的多尺寸 outcome 编排都在一次调用内，拆 turn 会让 async 占位/产物计数逻辑复杂化。
- B 让「确认」对模型透明：模型发起一次 `adapt_to_platform`，工具内部该等就等、该续就续，返回的还是完整 outcomes，无需模型参与确认编排。这与 `await_result`（同样在工具内阻塞等任务完成）的既有模式一致。
- 「不编辑自动继续」在 B 下是自然的：倒计时到期前端回传默认值，channel 收到即继续，对用户零点击。

代价：工具协程在等待期间占用一个 in-flight slot。用一个**带 ctx 取消 + 上限超时（如 15s）**的 select 兜底——前端 3s 倒计时正常情况下远早于此到达；若前端没回（旧客户端/掉线），超时后按 grok 原版默认继续，绝不无限挂起。

### Decision 2：3 秒计时放前端，后端只等 confirm 信号

`realtime-transport` spec 已确立「时间/队列编排归前端」（队列、插队、倒计时类节奏都在前端）。后端不该跑 3s 定时器——它只在 confirm channel 上 select{ 收到值 / ctx 取消 / 安全上限超时 }。

- 前端：分析块 `done` 折叠后进入确认态，启动 3s 倒计时。用户点「编辑」则**暂停倒计时**进入编辑；提交或倒计时到 0 时回传一次 confirm（编辑态提交携带编辑文本，到期携带原文）。
- 后端安全上限（15s）>> 前端 3s，仅用于前端缺席兜底，不参与正常路径。
- 幂等：confirm 对每个分析块只接收一次（前端提交后禁用面板；后端 channel 收一次即关闭/丢弃后续）。

### Decision 3：confirm 入站携带 cache key，回写覆盖原报告

前端回传 confirm 时携带该分析块对应的图片集 cache key（后端在进入确认态时下发，或前端原样回传分析事件里带的 key）。后端收到编辑文本后：

- 若文本与原版不同 → `InsertVisionReport(cacheKey, edited)` 覆盖，使同批图后续适配/改图（含 upload 预热命中）复用编辑版。
- 若文本未变（倒计时默认）→ 缓存中已是该版本，无需重复写。

`vision_reports` 此前只由 grok 写入；本次允许用户编辑覆盖。沿用「进程重启失效、下次重新分析」既有约定，不引入新的持久化语义。

### Decision 4：缓存命中也进入确认窗口；仅降级**不**进入

确认窗口在「本次有一份非空报告可供确认」时出现——无论该报告来自一次新的 grok 实时分析，还是来自缓存命中（预热版/此前编辑版）。两条路径共用同一门控逻辑（`gateSummaryConfirm`）：报告产出后、AI 重绘前暂停，用户可就地编辑，编辑后回写同一 cacheKey。

以下情况没有报告可确认，直接走既有降级路径、不弹确认：

- COS/vision 未配置、发布失败、分析失败、报告为空 → 既有降级（无 THEME 适配）。

理由：缓存命中时报告内容仍可能需要微调（例如换了适配场景、或上次编辑得不够好），用户应有同样的「过目即改」机会；改动回写后下次复用即拿到最新版。只有「根本没有报告」时弹确认才是噪音。

## Risks / Trade-offs

- **in-flight slot 占用**：等待期最长 8s（服务端安全上限）。缓解：硬上限超时 + ctx 取消（用户中断当前轮可立即解除）。
- **旧客户端不回 confirm**：超时后按原报告继续，体验等同今天（仅多一次最长 8s 延迟）。可接受，且属加法式兼容。
- **多块并发**：理论上一次 turn 内不会有两个分析块同时待确认（适配是单次工具调用、单个图片集）；confirm channel 按 session+cacheKey 寻址避免串扰。
- **用户编辑出格式**：不做结构校验，仅 `maxReportLen` 截断。错误格式至多让本次适配的 THEME 约束变弱，不影响系统稳定，符合「最小实现」。

## Migration Plan

加法式，无数据迁移。新入站类型与出站「进入确认态」标记对旧客户端透明（未知字段/事件忽略）。后端门控带超时兜底，即使前端完全不升级也能按原版默认继续。

## Open Questions

- 倒计时时长是否需可配？当前按用户要求固定 3s（前端常量），后端不感知具体秒数，后续若要可配只动前端。
