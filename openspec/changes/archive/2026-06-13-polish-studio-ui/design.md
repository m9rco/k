# Design — Polish Studio UI

## Context
前端是 `web/static` 下三文件，经 Go `embed` 提供服务（`cmd/server/main.go` 的 `spaHandler`）。本次优化以「设计师主导的体验打磨」为目标，但用户已明确**保持原生栈、不引入构建工具**。因此所有设计决策都要在「无 node 工具链、无打包步骤」约束下成立。

## Goals / Non-Goals
- Goals：科技感动效、关键路径流畅度、单图查看入口、拖拽排序+分栏、路线图降权、渠道标识、提示词优化。
- Non-Goals：Vite/框架迁移；抓取真实第三方商标；排序持久化；改生图/裁剪算法。

## Decision 1：前端栈——保持原生，仅做模块化重构
**选择**：不引入 Vite/框架。把 `app.js`（1384 行）按职责拆为多个 ESM 模块（如 `state.js` / `ws.js` / `workspace.js` / `capsule.js` / `prompt.js` / `ui.js`），用原生 `import` 组织；`styles.css` 抽出 `:root` 设计 token 分区。`index.html` 的 `<script type="module">` 已就绪，原生 ESM 多文件无需打包即可被 `http.FileServer` 逐个服务。

**理由**：当前规模（~2000 行）尚未到必须打包的体量；引入 node 工具链会带来构建步骤、依赖维护与 embed 流程改造成本，与「小团队内部工具」定位不匹配。模块化重构即可解决「单文件难维护」，且零运行时代价。

**权衡**：放弃了 HMR/tree-shaking/TS 类型检查等工程化收益；多文件会增加少量 HTTP 请求数（内网可忽略）。若未来规模翻倍，可在独立 change 中再评估升级（proposal 已记为 Out of scope）。

## Decision 2：科技感动效——CSS 优先，SVG 点缀，尊重 reduced-motion
- 动效以 CSS transition/animation 为主（GPU 友好的 transform/opacity），避免布局抖动。
- 品牌区与空状态用内联 SVG + CSS 动画（如描边流动、节点脉冲）营造科技感，体量小、无需额外资源请求。
- 统一过渡节奏（duration/easing 收敛为设计 token），保证「统一视觉语言」。
- 必须实现 `@media (prefers-reduced-motion: reduce)` 降级，关闭非必要动画（可访问性）。

## Decision 3：渠道视觉标识——首字母色块占位（不抓商标）
**选择**：为每个渠道生成「品牌色背景 + 首字/首字母」的色块标识，颜色取自前端维护的 `channelId → brandColor` 映射（找不到时按渠道分组取默认色）。在尺寸选择器的渠道列表项、已选项 chip 上展示。

**理由**：用户已确认不抓第三方真实 logo（版权 + 来源不稳定 + 部分渠道无公开素材）。色块占位零版权风险、纯本地、视觉统一，且后续若拿到合规素材可平滑替换为 `<img>`。

**实现位置**：渠道色映射作为前端常量（27 条），或在 `configs/channels.json` 既有结构上读取（当前无 color 字段，故先用前端常量，避免改后端契约）。

## Decision 4：单图查看入口与多选解耦
当前卡片左键 = toggle 多选，放大预览只在右键菜单里。新增一个**卡片内独立小按钮**（如右上角放大镜图标），`stopPropagation` 后直接 `openLightbox(asset)`，不影响左键多选语义。lightbox 已存在，复用即可。

## Decision 5：工作区拖拽排序 + 多列分栏（前端展示态）
- **分栏**：「已完成」阶段的卡片网格从当前布局改为多列响应式网格（窄屏退化为少列/单列），保留里程碑分组（进行中/已完成/失败）。
- **排序**：仅对「已完成」资产支持拖拽调序，用原生 HTML Drag and Drop（draggable + dragover/drop）。顺序保存在前端 `state` 的一个 `order: assetId[]` 数组里，`renderWorkspace` 按该数组排序渲染。**不持久化到后端**（用户已确认展示态即可）；刷新后回退到后端返回的默认顺序（newest-first）。
- **权衡**：HTML5 DnD 在触摸端体验一般，但本工具以桌面为主，可接受；不引入第三方拖拽库以维持零依赖。

## Decision 6：路线图视觉降权
将右栏常驻的「规划中的能力」大卡片列表，改为**紧凑/可折叠**呈现（默认收起或移到工作区底部的细条目入口），把主视觉让给工作区资产。spec 的「展示规划能力」要求仍满足（仍可见、仍标注「规划中」、仍不触发后端），只是占位收敛。

## Decision 7：提示词优化端点
**选择**：新增 `POST /api/session/{id}/prompt/optimize`，请求体 `{ "text": "<口语化输入>" }`，响应 `{ "optimized": "<改写后的生图提示词>" }`。服务端用会话理解模型（claude-sonnet-4-6）以一个**独立的改写指令**把口语化描述结构化为生图提示词，**仅返回文本，不调用任何生图/工具、不写入会话 context、不影响滑动窗口**。前端点击「优化」按钮 → 调用端点 → 把结果回填到输入框（替换或追加），用户确认后再正常发送。

**理由**：
- 用户已选「新增后端优化端点」，质量优于纯前端模板，且不占用一轮对话（与「复用对话 Agent」相比不污染 context）。
- 独立端点便于约束：改写指令明确「只输出提示词、不解释、不执行」，并复用现有 prompt-injection 防护原则（把用户文本当数据，不当指令）。

**安全 / 健壮**：
- 输入长度上限，超长截断；空输入直接返回原文。
- 模型不可用 / 超时：返回明确错误，前端 toast 提示且保留用户原始输入不丢失。
- 该端点不进入白名单意图执行路径，纯文本改写，不触发素材产出。

## Risks
- 模块拆分可能引入跨文件状态耦合 bug：通过集中 `state` 模块 + 明确导出边界降低风险，重构后跑通现有交互回归。
- 动效过度导致性能/眩晕：用 reduced-motion 降级 + 仅 transform/opacity 动画约束。
- 提示词优化端点被滥用为通用对话：靠改写指令约束 + 长度上限 + 不接工具来限制。
