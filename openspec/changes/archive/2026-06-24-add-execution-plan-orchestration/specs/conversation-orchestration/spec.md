## ADDED Requirements
### Requirement: 复合多步请求的计划分发
系统 SHALL 将 `submit_plan` 纳入 Agent 可用工具集合。当用户一句话请求包含**多个有产物依赖的连续操作**（如「换角色然后切尺寸」「找图再生视频」「做成 N 个尺寸」）时，系统 SHALL 引导 Agent 通过 `submit_plan` 一次性提交完整有序步骤，交由服务端串行执行；而非让 Agent 仅调用单个工具后停止、或自行多轮手动串联。对于**单个操作**的请求，Agent SHALL 直接调用对应单工具，SHALL NOT 为单步请求生成计划。该引导与既有"命中白名单即直接调工具、不先发文字确认"的核心规则一致。

#### Scenario: 复合请求分发到 submit_plan
- **WHEN** 用户请求「第二张换成第一张的角色，再做成 iOS 4 个尺寸」
- **THEN** Agent 识别为多步依赖请求并调用 submit_plan 提交完整步骤
- **AND** 不只调用 edit_image 就停止、也不漏掉后续切尺寸步骤

#### Scenario: 单步请求不生成计划
- **WHEN** 用户只要求「换个背景」
- **THEN** Agent 直接调用 edit_image，不调用 submit_plan

### Requirement: 复合请求的确定性预分类
确定性意图预分类 SHALL 识别复合多步信号（如「然后/接着/再/并/同时/做成…尺寸/各平台/N 个尺寸」等连接词或多目标表述，且文本命中 ≥2 个白名单动作）。命中时系统 SHALL 在意图提示中追加"建议用 submit_plan 一次提交完整步骤由系统串行执行"的引导。该提示 SHALL 仅供参考、会话模型仍是最终决策者（与既有意图提示同构，且只作数据、不作可执行指令）。

#### Scenario: 多步信号命中追加计划提示
- **WHEN** 用户文本含连接多动作的表述（如「换角色…然后…各平台尺寸」）且命中至少两个白名单动作
- **THEN** 预分类在意图提示中追加建议使用 submit_plan 的引导
- **AND** 该引导仅供参考，模型可据真实理解仍选择单工具或不同分解

#### Scenario: 单动作文本不触发计划提示
- **WHEN** 用户文本仅命中单一白名单动作且无多步连接信号
- **THEN** 预分类不追加 submit_plan 引导
- **AND** 既有单步意图提示与直调行为不受影响

### Requirement: 计划执行规范纳入分层 System Prompt
分层 system prompt 的工具使用规范 SHALL 包含计划编排说明：复合多步且后步依赖前步产物的请求，用 `submit_plan` 一次提交完整有序步骤（每步含工具名、参数、用 `$stepN.asset_id` 占位符引用前序产物），由系统串行执行并在失败时立即中断；单步请求直接调对应工具。该说明 SHALL 给出至少一个含占位符的步骤示例，并 SHALL NOT 削弱既有的同轮工具调用、白名单拒绝、澄清优先、再次请求即再次执行、简体中文回复等约束。

#### Scenario: prompt 含计划编排与占位符示例
- **WHEN** 构建 system prompt
- **THEN** 工具使用规范含"复合多步依赖请求用 submit_plan 提交完整步骤、用 $stepN.asset_id 引用前序产物、由系统串行执行、失败立即中断"的说明与示例

#### Scenario: 既有约束保留
- **WHEN** 构建 system prompt
- **THEN** 同轮工具调用、白名单礼貌拒绝、澄清优先、再次请求即再次执行、简体中文回复等既有约束均保留
