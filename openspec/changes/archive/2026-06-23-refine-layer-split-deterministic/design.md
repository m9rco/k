# Design — 分割掩码抠图分层

## Context

「图层精修」当前用 AI 重绘做分层（详见 proposal.md「Why」）。用户的真实诉求是**人工微调**，需要忠实原图、可移动、**透明底**、数量克制的图层。本设计把分层机制从「两次 AI 重绘」改为「**零生图调用的分割掩码抠图**」：视觉模型只返回主体的**分割掩码**，服务端把掩码作为 alpha 贴到原图原始像素上。三个用户决策已锁定：抠图=分割掩码（透明底、保真、不重绘）、背景=原图不重绘、分层=人物+宣发文案两类。

> 注：本 change 早期版本曾用「按框矩形裁切」（主体层为不透明矩形块）。用户反馈矩形块「跟透明底没关系、不是真正分割图层」，故修订为分割掩码抠图——既透明底、又保持像素来自原图（见 D2）。矩形裁切作为**掩码缺失时的降级路径**保留。

## Goals / Non-Goals

**Goals**
- 主体层 RGB 100% 来自原图，叠回原位与原图严丝合缝（解决「连接不上」）。
- 主体层为**透明底抠图**（掩码作 alpha），移走主体背后干净露出背景层（解决「跟透明底没关系」）。
- 背景层=原图本身，稳定、零等待、零失真（满足「底线固定」）。
- 检测只产出人物 + 宣发文案两类、上限 5（满足「不要过多元素」）。
- 分层不调用任何生图大模型（更快、更稳；掩码是分析输出，非生图）。

**Non-Goals**
- 不用 AI 重绘主体（绿幕重画 + 键绿）做抠图——重画与原图对不上，正是「连接不上」的根因。透明底改由「分割掩码 × 原始像素」实现。
- 不删除 `extract_layer`/`fill_background` 意图与 chroma-key（保留给未来手动精确抠图或其它入口）。
- 不改合成导出 / 持久化端点 / 时间轴归类。

## Decisions

### D1：背景层 = 原图 asset（移除 fill_background）
分层结果的锁定底层直接引用源图 `assetID`，不再 spawn `fill_background`。
- **理由**：用户要「底线固定」。原图是唯一稳定、零成本、零失真的底层。AI 重绘背景每次不同且偶发失败——与「固定」语义直接冲突。
- **取舍**：主体层被移走后，底下露出的是原图里该主体的原貌（而非补全的干净背景）。在「人工微调」语义下可接受——用户移动主体多是小幅位移/缩放，不会大面积裸露；真要干净背景可单独走 `fill_background`（保留的意图）。

### D2：主体层 = 分割掩码 × 原图原始像素（透明底抠图，移除 extract_layer）
对检测出的每个主体，视觉模型返回与其包围盒对齐的**分割掩码**；服务端按包围盒在原图上取出**原始像素、不缩放**的子图，把掩码缩放对齐后作为 **alpha 通道**写入——RGB 保持原图、掩码之外透明，产出透明底抠图层。
- **实现**：`vision` 侧 `Subject` 增 `Mask []byte`，Gemini 传输用 `subjectMasksPrompt`（`box_2d` 0..1000 + `mask` base64 PNG 概率图），`parseSubjectMasks` 解码；非 Gemini 传输无掩码能力，仍走 box-only（`Mask=nil`）。`layering/crop.go` 的 `cropSubject(src, box, mask)`：掩码存在时按框精确取子图（无 padding——掩码即边界），用 `golang.org/x/image/draw` 把灰度掩码缩放到子图尺寸，逐像素 `alpha = srcAlpha * mask/255`，RGB 不动；PNG 编码。
- **降级**：掩码缺失/不可解码时回退到原「padding + 不透明矩形」裁切（像素仍 100% 来自原图），保证分层不失败。
- **位置回传**：每层返回其子图框的**归一化左上角** `{x,y}`，前端据此把子图落在原位 → 初始与原图重合。
- **理由**：RGB 来自原图 → 零错位、零漏边、零染色、零多余轮廓（根治「连接不上」）；掩码只控透明度 → 真正透明底抠图（解决「跟透明底没关系」）。掩码是分析输出（与包围盒同性质），不触碰任何生图大模型。
- **取舍**：掩码质量依赖视觉模型分割能力；复杂边缘可能有少量软边/毛刺，可接受（微调场景）。无掩码能力的模型只能得到不透明矩形（降级）。

### D3：检测两类化 + 上限 5
`subjectsPrompt` 改为只列「① 每个角色/人物 ② 每个宣发文案块（标题/卖点/活动文案）」；显式排除 LOGO、道具/物件、场景、天空、地面、装饰底纹（全部留背景）。`maxDetectedSubjects` 8→5。
- **理由**：直接对齐用户「只要人物 + 宣发文案、不要过多元素」。
- **取舍**：主推商品/吉祥物这类「核心物件」若不被视作角色，会留在背景层不可单独微调。用户已确认接受（选项一）。

### D4：layer-split 端点同步化（移除 await 轮询）
`Service.Split` 不再 `gen.Start` + 轮询任务，改为：读原图字节 → `DetectSubjects` → 对每个 subject `cropSubject(img, box, mask)` 抠图落库为 asset → 背景层引用源图 → 返回有序图层（背景最底 + 主体按重要性）。
- **落库**：裁切子图复用 `composite` 的持久化路径（或新增 `layer` kind）落为当前会话 asset，复用无损 PNG 优化。倾向沿用既有 asset 落库 helper，kind 用 `composite`（已存在、时间轴已归类【拼接】），避免新增 kind 牵动前端枚举。**待定**：若希望时间轴把「裁切层」与「合成产物」区分，再引入 `layer` kind；本期从简复用 `composite`。
- **理由**：无生图依赖后，编排退化为纯本地图像处理，可同步完成（毫秒级），删去 generator 接口耦合与 await 复杂度。
- **取舍**：`layering` 不再依赖 `generation`；其 `generator` 接口与 `await`/`poll` 字段移除。`fill_background`/`extract_layer` 的路由代码留在 generation（停用不删）。

### D5：前端按 Box 摆放
`SplitLayer` 增加 `box {x,y,w,h}`（归一化）。背景层 box=全画幅 `{0,0,1,1}`、`x:0,y:0`。主体层 `x = box.x*canvasW, y = box.y*canvasH`，`scale:1`。导出逻辑已按 `naturalWidth*scale` 绘制、按 `x/y` 落位——天然兼容子图层（之前整图层恰好 box 全画幅、x/y=0，是本方案的特例）。
- **理由**：裁切层是主体大小的子图，必须按原位摆放才能初始重合；统一 x:0/y:0 只对「整图等尺寸层」成立。

## Risks / Trade-offs

- **检测/掩码偏差**：框或掩码偏紧裁掉主体边 / 偏松带背景。→ 透明底下偏松的背景被掩码切掉；偏紧由用户微调时移动/缩放吸收。检测失败（空列表）仍按现有逻辑报「未检测到可分层主体」。
- **掩码软边/毛刺**：复杂边缘掩码概率图可能有软边。→ 软 alpha 反而边缘自然，可接受。
- **无掩码能力模型**：非 Gemini 传输无分割能力 → 降级为不透明矩形（见 D2 降级），不失败但非透明底。
- **露底是原图而非干净背景**：见 D1 取舍，符合微调语义；保留 `fill_background` 意图作为未来可选增强。
- **复用 `composite` kind**：裁切层与最终合成图同 kind，时间轴都归【拼接】。若造成混淆，后续可引入 `layer` kind（本期不做，YAGNI）。

## Migration / Rollout
- 纯行为替换，无数据迁移。旧的 AI 重绘分层产物已落库的 asset 不受影响（它们只是普通 asset）。
- `extract_layer`/`fill_background` 单测保留；layering 单测改写为断言裁切结果。
