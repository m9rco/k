# conversation-orchestration — Delta

## MODIFIED Requirements

### Requirement: 流式对话输出
系统 SHALL 以**真·流式**方式消费模型供应商的流式响应，并将 agent 的回复增量与思考增量分别实时推送给前端，而非先获取完整结果再切块模拟。当模型返回思考内容（reasoning/thinking）时，系统 SHALL 将其作为独立于回答正文的增量类型推送；对于支持显式开启思考的会话供应商（如 Anthropic extended thinking），系统 SHALL 在请求中开启思考能力，使思考增量得以产生并下发，从而保证前端的思考块有内容可逐字渲染；当供应商流式解析失败而降级为读取完整响应后补发时，系统 SHALL 同样将思考内容**按增量分片**补发（而非整段一次性下发），保持打字机式呈现一致。此外，系统 SHALL 在收到用户消息后、调用模型之前立即发出一个"轮开始"信号，并在该轮结束（含错误、产出反问或被中断）时发出"轮结束"信号，使前端的 loading 态与模型首个增量解耦。当一轮既无回复正文、又无工具调用、又无反问产出时，系统 SHALL 在轮结束信号中标明该轮无正文，使前端可避免渲染空白回复气泡。

#### Scenario: 回答增量流式推送
- **WHEN** agent 生成回复
- **THEN** 系统按供应商返回的增量片段实时推送，前端可逐步渲染而非等待完整结果

#### Scenario: 思考增量独立推送
- **WHEN** 模型在回答前返回思考内容
- **THEN** 系统将思考增量以区别于回答正文的事件类型逐片推送
- **AND** 前端据此实时渲染思考块

#### Scenario: 开启供应商思考能力
- **WHEN** 当前会话供应商支持显式开启思考（如 Anthropic extended thinking）
- **THEN** 系统在模型请求中开启思考能力并设置合理的思考预算与输出上限
- **AND** 由此产生的思考增量经既有思考事件类型实时下发给前端

#### Scenario: 流式失败降级时思考仍分片
- **WHEN** 供应商流式响应解析失败，系统降级为读取完整响应后补发
- **THEN** 系统将完整响应中的思考内容按增量分片补发，而非整段一次性下发
- **AND** 前端的思考块仍呈逐字涌现，不出现整段突现

#### Scenario: 空轮在轮结束标明无正文
- **WHEN** 一轮处理结束且既无回复正文、又无工具调用、又无反问
- **THEN** 系统在轮结束信号中标明本轮无正文
- **AND** 前端据此不渲染空白回复气泡

### Requirement: 面向 Web 的输出格式规范
系统 SHALL 在 system prompt 中倡导 agent 的文本回复为面向 web 渲染的简洁文本，需要让用户在多个具体取值之间做选择时优先通过澄清工具产出结构化选项而非文本编号列表。当 agent 仍然输出了 markdown 标记时，前端 SHALL 将其**渲染**为对应的富文本呈现，而非剥离标记或将 markdown 源码原样暴露给用户；面向用户应说明的内容 SHALL 写入正式回复正文，不得仅写在思考区。

#### Scenario: 优先结构化选项
- **WHEN** agent 需要用户在多个具体取值间选择
- **THEN** agent 优先通过澄清工具产出结构化选项

#### Scenario: markdown 被渲染而非暴露
- **WHEN** agent 的回复正文包含 markdown 标记
- **THEN** 前端将其渲染为对应富文本（如加粗、列表）而非显示 `**`、`#` 等源码
- **AND** 不将 markdown 源码原样暴露给用户

#### Scenario: 面向用户内容写入正文
- **WHEN** agent 有需要让用户看到的说明（包括对不支持请求的礼貌拒绝）
- **THEN** 该内容出现在正式回复正文中，而非仅存在于思考区
- **AND** 用户能在对话区看到完整回复，不出现"思考完却无回复"的断裂

### Requirement: 分层 System Prompt
系统 SHALL 以分层结构组织 agent 的 system prompt，至少包含：角色定义、能力白名单、工具使用规范、交互与澄清规范、输出格式规范、安全规范、语言要求。其中 SHALL 包含"意图缺少必要信息时先澄清再行动"的指令，并保留既有的同轮工具调用、白名单拒绝、未配置告警、引用 id 使用、prompt 注入防护、简体中文回复等约束。语言要求 SHALL 同时覆盖**思考过程**（thinking/reasoning 也使用简体中文）。工具使用规范 SHALL 教导 agent 借助注入的"资产编号映射"理解用户以"图N"指代的资产，并区分多图意图：当用户表达"根据图X图Y生成/创作新图"时以图X图Y 作为参照（reference）且不设被编辑底图（source）；当用户表达"把图X图Y放进/融合到图Z"或"在图Z基础上修改"时以图Z 作为被编辑底图（source）、图X图Y 作为参照（reference）。面向用户的回复 SHALL 写入正文且可在说明性文字中以"图N"称呼资产。

#### Scenario: prompt 含澄清优先指令
- **WHEN** 构建 system prompt
- **THEN** prompt 含"意图缺少必要信息时应先调用澄清工具反问而非猜测"的指令

#### Scenario: 既有约束保留
- **WHEN** 构建 system prompt
- **THEN** 同轮工具调用、白名单礼貌拒绝、未配置告警、引用 id 使用、prompt 注入防护、简体中文回复等既有约束均在对应层中保留

#### Scenario: 思考过程语言为中文
- **WHEN** 构建 system prompt
- **THEN** 语言要求明确思考过程（thinking/reasoning）也使用简体中文

#### Scenario: 多图意图区分参照与编辑目标
- **WHEN** 构建 system prompt
- **THEN** 工具使用规范含示例：「根据图X图Y生成新图」→ 图X图Y 为 reference、无 source；「把图X图Y放进图Z」→ 图Z 为 source、图X图Y 为 reference

#### Scenario: 据编号映射解析"图N"
- **WHEN** 用户消息以"图N"指代资产且上下文已注入编号映射
- **THEN** agent 据映射将"图N"解析为对应 asset_id 并填入工具参数

## ADDED Requirements

### Requirement: 对话轮中断
系统 SHALL 支持中断一个正在进行的对话轮：当收到针对当前 session 的中断请求时，系统 SHALL 取消该轮正在进行的模型推理与工具循环，停止继续下发该轮增量，并发出标记为"已中断"的轮结束信号。中断 SHALL 仅作用于对话轮的模型推理，不取消已提交的异步生图/生视频任务（这些任务有独立的取消入口）。同一 session 的对话轮 SHALL 串行处理，系统 SHALL 保证"取消旧轮 → 开始新轮"的有序切换，避免两轮交叠写入会话上下文。

#### Scenario: 中断进行中的轮
- **WHEN** 某 session 有一个对话轮正在进行，系统收到该 session 的中断请求
- **THEN** 系统取消该轮的模型推理与工具循环并停止下发其增量
- **AND** 发出标记为"已中断"的轮结束信号

#### Scenario: 中断不影响已提交的异步任务
- **WHEN** 对话轮中断时已存在已提交的生图/生视频任务
- **THEN** 这些异步任务不被该中断取消，仍按各自流程进行

#### Scenario: 取消旧轮后有序开始新轮
- **WHEN** 用户在一轮进行中要求打断并立即处理新消息
- **THEN** 系统先取消旧轮再开始新轮，两轮不交叠写入会话上下文
