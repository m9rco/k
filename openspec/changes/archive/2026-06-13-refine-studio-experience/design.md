## Context
体验优化专项横跨前端布局/交互、对话编排（真流式）、生图（尺寸继承）、下载（分目录）四块。其中"真·逐字流式思考"会改动 `chatmodel.go` 的传输契约（当前是"先 `Generate` 拿全量再切块"的伪流式），属于跨层、有回归风险的变更，需要在动工前固化技术决策。其余多为前端体验改造，但"操作目标一致性"是正确性 bug，需明确根因与防回归约束。

## Goals / Non-Goals
- Goals:
  - 把对话区确立为产品核心：布局重心、工具调用呈现、状态严控、思考可见。
  - 让"所见即所操作"：选中/右键/二次调整严格作用于目标资产。
  - 长任务等待不再是黑盒：阶段化进度 + 可继续操作。
  - 补齐切尺寸（全选、批量、分目录）与尺寸继承能力。
- Non-Goals:
  - 不实现自学习/团队知识库/生视频的真实后端能力（需求6 仅前端展示规划）。
  - 不引入鉴权、用户系统（沿用匿名 session）。
  - 偏好分析不做后端 NLP/模型调用（需求5 走前端关键词提取）。

## Decisions

### D1: 思考与回复采用 provider 真流式（需求1）
- **决策**：将 `chatModel.Generate/Stream` 改为消费 provider 的 SSE 流（OpenAI 兼容 `stream:true` 的 `delta.content` / `delta.reasoning_content`；Anthropic `messages` 流式的 `content_block_delta` 含 `thinking`/`text`）。`agent.go` 的流式循环逐帧把 `reasoning` 增量与 `content` 增量分别 emit。
- **理由**：用户明确选择"真·逐字流"。伪流式无法体现"思考逐字涌现"的体感，且思考与回答的时序在伪流式下被压平。
- **Alternatives**：保留伪流式仅前端做打字机动画——被否，思考块的真实时序（先想后答、想-调工具-再想）无法还原。
- **回退**：流式解析失败时降级为一次性读取全量响应后按帧补发，保证不空屏。

### D2: 思考块折叠态（需求1）
- **决策**：思考增量进入一个"思考中…"可展开容器，逐字追加；当本轮首个回答 `content` 帧到达（或工具调用开始）即把该容器置为折叠态，保留展开入口。每个 ReAct 步可有独立思考块。
- **理由**：思考是过程信息，回答是结论；产出结论后默认收起，避免长思考淹没对话。

### D3: 操作目标一致性（需求3）
- **决策**：资产卡片的所有操作入口（左键选中、右键菜单、二次调整、下载、切尺寸）均以**显式 assetId** 绑定到被操作的具体卡片，不依赖"当前活动资产"这类隐式单例状态在异步流程中被覆盖。`activeAssetId` 仅作为发消息时的一次性 ref，设置与消费之间不得插入其他可改写它的路径。
- **理由**：根因是隐式 `activeAssetId` 在多卡片/异步场景被后续操作覆盖，导致"选第三张、操作第一张"。改为每个操作闭包捕获自身 assetId 可根除。
- **防回归**：spec 增加"操作目标与所选资产严格一致"的 Scenario 作为验收基准。

### D4: 二次调整尺寸继承（需求12）
- **决策**：当 `SourceAssetID` 非空时，生图 `Request.Width/Height` 取源资产的原始宽高（来自 store 的 `AssetRecord.Width/Height`），透传给 provider；provider 现有 `sizeParam` 将其 snap 到最接近的受支持尺寸 enum，并按需要在产物落库时记录实际宽高。
- **理由**：用户期望"改完还是原图那么大"。当前 `Request` 不带尺寸，永远走默认 1024²，导致二次调整后尺寸跳变。
- **取舍**：gpt-image-1 仅支持固定 enum，无法逐像素等尺寸；采用"snap 到最接近 enum + 记录真实尺寸"的务实策略，spec 用"尽量与源图一致"措辞而非"逐像素相等"。

### D5: zip 分目录（需求9）
- **决策**：批量打包时按 `渠道/尺寸` 维度组织 zip 内目录结构（如 `taptap/icon-512/asset.png`）；裁剪产物落库时需携带其渠道与尺寸来源信息以供分目录。源/无尺寸归属的资产归入 `misc/` 或按 kind 分目录。
- **理由**：用户拿到 zip 后要按平台交付，平铺命名难用。

### D6: 偏好关键词提取（需求5）
- **决策**：前端在发送用户消息时做轻量分词/关键词计数（停用词过滤 + 高频统计），累计在前端状态并渲染到偏好角落；无任何关键词时角落 `hidden`。不落后端。
- **理由**：用户选择前端关键词提取，零后端成本、即时反馈。

### D7: 布局比例 3:7（需求11）
- **决策**：主布局 `grid-template-columns` 改为工作区:对话 = 3:7（对话区在视觉与空间上为主）。需重新确认左右位置：对话区为主区域。响应式窄屏维持上下堆叠。

## Risks / Trade-offs
- **真流式回归风险（高）** → 两家 provider 流式协议不同，需各自契约测试；保留 D1 的降级路径。
- **gpt-image-1 尺寸 enum 限制** → 无法严格等尺寸，spec 措辞放宽为"贴近源图"，并记录实际产物尺寸。
- **裁剪产物渠道/尺寸归属缺失** → 分目录依赖该元数据，需确认 crop 产物落库时已带或补齐渠道+尺寸来源。

## Migration Plan
1. 后端先上真流式（D1）+ 尺寸继承（D4）+ 分目录（D5），各自带契约/单测，灰度验证不空屏、尺寸贴近、目录正确。
2. 前端布局比例（D7）与目标一致性修复（D3）优先（可见且低风险）。
3. 思考折叠（D2）、工具状态卡、偏好角落（D6）、路线图卡、长任务反馈依次叠加。
4. 全程 `go test ./...` + 浏览器手测三类交互回归。
