# Design: gpt-image-2 多参考图低发散适配 harness

## Context

核心诉求（用户原话提炼）：**给 1~16 张参考图，产出构图和谐的各目标尺寸图片；务必保留核心图片的内容与意图，不发散、不随意发挥；和谐是第一要素。**

约束来源（OpenAI 官方 image-generation 文档，2026-06 抓取核实）：

- gpt-image-2 的 `size` **不是固定枚举**，接受满足约束的任意分辨率：最长边 ≤3840px、两边均为 16 的倍数、长短比 ≤3:1、总像素 ∈ [655360, 8294400]；>3686400px（>2K）为实验性。`auto` 为默认。
- gpt-image-2 **不接受 `input_fidelity` 参数**——模型自动对每张输入图按高保真处理（代价是 edit 请求输入 token 升高）。
- 多参考图经 edits 端点 `image[]` 数组传入；带 mask 时 mask 仅作用于第一张图。
- gpt-image-2 **不支持透明底**（`background: "transparent"` 无效）。
- `quality`: `low/medium/high/auto`；草稿用 `low`。

现状偏差：`internal/generation/http_provider.go` 的 `legalGenSizes` 写死 gpt-image-1 的 3 档；`MaxReferenceImages=6` 且参考图无角色分层；prompt 模板未把「保留/不发散」结构化；透明底等不可满足约束被原样注入。

**目标档位约束（核实 `configs/channels.json` 全部 139 个 producible 档位 vs gpt-image-2 约束，2026-06）**：

| 边界 | 命中档位数 | 占比 | 处置 |
|---|---|---|---|
| 总像素 < 655360（低于出图下限） | **79** | **57%** | **放大同比例生成 + 下采样收敛（主路径）** |
| 长短比 > 3:1（超约束） | 11 | 8% | 夹到 3:1 合法比例生成 + `cover` 收敛 |
| 总像素 > 3686400（>2K 实验档） | 2（iOS 2732×2048） | 1.4% | 夹 2K 或放行实验档（见 Open Q） |
| 透明底（gpt-image-2 不支持） | 2（B站 logo / 虎牙礼包 150） | 1.4% | 约束改写（D4） |

**硬约束（用户明确）：产物尺寸必须精确等于目录档位，绝不偏离。** gpt-image-2 出图尺寸只是"中间生成尺寸"，最终一律由收敛步骤压到目录档位的精确像素。因此：**>半数档位的真实路径是「按目标比例放大到 gpt-image-2 合法区间出图 → 等比下采样到目录精确尺寸」**——这反而对成片质量有利（小图由更高分辨率下采样，更锐利）。最极端的 `1008×168`(6:1)、`480x108`(4.44:1) 等 banner 走「夹 3:1 出图 + cover 裁到精确档位」。

## Goals / Non-Goals

**Goals**
- 在 gpt-image-2 约束内**直接命中目标比例**生成，把收敛形变降到最低（和谐优先）。
- 1~16 张参考图，**锚点图锁定内容与意图**，辅助图只贡献风格/元素。
- prompt 把「和谐、保留、不发散」编码为模型可执行的结构化指令。
- 不可满足的平台约束（透明底等）不诱发幻觉。

**Non-Goals**
- 不引入厂商 SDK（沿用手写 HTTP 适配器约定）。
- 不改裁剪快路径（比例一致仍走确定性裁剪，不调 AI）。
- 不做 mask/inpainting（本期仍为整图重绘）。
- 不改会话模型选择 / 适配请求级路由（gpt-image-2 固定路由已在 platform-adaptation 既有需求中）。

## Decisions

### D1. gpt-image-2 尺寸解析：从「三档就近」改为「约束内贴合目标比例」

**问题**：旧 `nearestGenSize` 把任意目标压到 3 个比例之一，1080×1920(9:16) 被压到 1024×1536(2:3)，收敛步骤再裁/补一次 —— 两次形变，构图被动改变，和谐度下降。

**方案**：新增 gpt-image-2 专属解析器 `resolveGptImage2Size(dstW, dstH int) string`，返回**贴合目标比例、落在 gpt-image-2 合法区间**的中间生成尺寸（最终精确档位由收敛步骤保证，见 D5）。算法：

1. 计算目标长短比 `r = max/min` 与方向。
2. **确定生成比例**：
   - `r ≤ 3.0`：用**目标原始比例**；
   - `r > 3.0`（极端横/竖幅）：把比例**夹到 3:1**（gpt-image-2 上限），由收敛 `cover` 裁到精确目标。
3. **确定生成像素量**：以「质量优先」选一个**目标像素预算**（默认贴近但不超过 2K：约 2_000_000~3_600_000px，规避实验档），即使目录档位很小（57% 档位 <655360px），也按该预算放大出图——既满足下限，又让下采样后的小图更锐利。
4. 按选定比例 + 像素预算反解 W、H：两边各 round 到最近 16 倍数、clamp 最长边 ≤3840、并夹总像素入 [655360, 8294400]。
5. **>2K 档位**（iOS 2732×2048）：按目录判定是否放行实验档（见 Open Q），否则夹 2K 出图后由收敛上采样到精确档位。
6. 任一维度为 0（纯文生图未指定）：返回空 → 走供应商默认（`auto`）。

**为何不用 `auto`**：`auto` 让模型自选尺寸，输出比例不可控，收敛步骤无法预估裁/补 —— 与「和谐优先 + 精确档位」相悖。显式传贴合比例的合法尺寸保持确定性。

**为何按适配器隔离**：`sizeParam` 当前被 `http_provider.go` 与 `dashscope.go` 共用，但二者合法尺寸完全不同（DashScope 是 `1024*1024` 等固定枚举，Gemini 走 `aspectRatio` 根本不传 size）。把 gpt-image-2 解析逻辑收敛进 gpt-image 适配器内部，DashScope 保留自己的枚举映射，Gemini 不变 —— 避免一处改动污染其他供应商。

| 目录档位（必须精确产出） | r | 旧行为（3 档就近 + 收敛） | 新行为（gpt-image-2 贴合比例 + 收敛到精确档位） |
|---|---|---|---|
| `1280×720` 16:9 | 1.78 | 1536×1024(3:2) → 裁/补 | 同 16:9 放大出图 → 下采样到 1280×720 |
| `720×1280` 9:16 | 1.78 | 1024×1536(2:3) → 裁/补 | 同 9:16 放大出图 → 下采样到 720×1280 |
| `512×512` icon | 1.0 | 1024×1024 → 下采样 | 1:1 放大出图 → 下采样到 512×512 |
| `900×600` cover (<下限) | 1.5 | 1536×1024 → 裁/补 | 同 3:2 放大出图 → 下采样到 900×600 |
| `1120×280` banner 4:1 | 4.0 | 1536×1024 → cover | 夹 3:1 出图 → cover 裁到 1120×280 |
| `1008×168` banner 6:1 | 6.0 | 1536×1024 → cover | 夹 3:1 出图 → cover 裁到 1008×168 |
| `2732×2048` iOS (>2K) | 1.33 | 1536×1024 → 上采样 | 4:3 贴近 2K/实验档出图 → 收敛到 2732×2048 |

### D2. 参考图角色分层：锚点 + 辅助（1~16）

**问题**：扁平「主参考 + 附加」无法表达「内容真相源」与「风格灵感源」的差别；多图最常见失败是模型把多张图**平均**成新主体（发散）。

**方案**：
- `MaxReferenceImages` 6 → 16；超 16 截断为前 16 并提示（沿用既有截断语义）。
- 第一张 = **锚点图（anchor）**：主体身份、核心构图、宣发意图的**唯一真相源**；调色板提取、尺寸继承、parent 链接均以它为准（保持现状）。
- 第 2~16 张 = **辅助参考（auxiliary）**：仅供风格、配色、元素、构图灵感；**不得改写锚点主体的身份或核心信息**。
- prompt 模板显式声明该契约（见 D3），并按顺序在 `image[]` 透传（锚点为 `image`，辅助为 `image[]` 其余项 —— 沿用 gpt-image 多图约定）。
- 供应商不支持多图时降级为仅锚点图 + 文字描述其余（保持既有降级语义）。

**为何 16 不再分页/分批**：gpt-image-2 单请求即支持多图数组，16 在文档示例量级内；超量截断而非报错，符合既有「礼貌降级」基调。

### D3. 低发散 prompt harness：上下文 / 保留 / 修改 / 禁止 四段式

**问题**：现有模板对各意图写了正向指令，但「不发散」靠零散措辞（"Preserve the existing scene"），缺少 ① **领域上下文**（模型不知道这是游戏宣发素材，容易当通用插画自由发挥、虚构游戏里不存在的画面）；② **统一的负向约束**；③ **锚点角色声明**。多图/换比例时尤其容易跑偏。

**方案**：所有图生图意图（`change_*`、`add_character`、`generate_icon`、`adapt_platform`）在服务端模板**最前置** `[CONTEXT]` 领域声明，并在尾部统一拼接 `[PRESERVE]/[MODIFY]/[AVOID]` 三段（文生图无源图，仍可加轻量 `[CONTEXT]`，但不加 PRESERVE）：

```
[CONTEXT]  This is a promotional marketing asset for an existing video game. The reference
           image(s) are authentic key art / in-game visuals from THAT game. Stay strictly
           within the game's established art style, world, characters and tone. Do NOT
           invent gameplay, UI, scenes, characters or worlds that are not present in the
           reference; do NOT turn it into generic fantasy/stock illustration.
[PRESERVE] 保留锚点图的主体身份（角色样貌/服装/配色/标志性特征）、核心宣发信息、关键视觉元素，忠于其表达意图。
[MODIFY]   <本意图的具体修改指令，既有 per-kind 文案 + sanitized 用户 slot>
[AVOID]    不要发明新主体、新场景或游戏中不存在的画面；不改变角色身份；不新增锚点图中没有的文案/水印/logo；
           辅助参考图仅供风格与元素借鉴，不得替换锚点主体；保持整体色调与光照和谐，避免突兀对比。
```

- `[CONTEXT]` / `[PRESERVE]` / `[AVOID]` 为**服务端固定文案**，不接受用户文本，杜绝注入改写。
- `[MODIFY]` 承接既有 per-kind 指令 + sanitized 用户 slot（既有注入防护不变）。
- 既有 `harmonyConstraint` 与 palette 约束并入 `[AVOID]/[PRESERVE]` 语义，不重复。
- 多参考图（≥2）时，`[PRESERVE]` 追加一句显式锚点声明：「第一张为锚点图，是内容与意图的唯一真相源」。

**游戏宣发「不虚构」的三层抓手**（直接回应"告诉它是游戏宣发、不要脱离游戏本身/不要虚构游戏画面"）：
1. **领域定性**（`[CONTEXT]`）：明确"现有游戏的宣发素材""参考图是该游戏真实美术"，把模型从"通用插画师"切换到"该游戏美术延展"。
2. **真相源锚定**（`[PRESERVE]` + 自动高保真，见 D4）：锚点图是唯一内容真相源，gpt-image-2 自动按高保真读入，主体细节天然不漂。
3. **负向围栏**（`[AVOID]`）：逐条点名最常见的发散模式——造新场景/角色/UI、改身份、加未授权文案、多图平均化——做定点封堵。

**为何四段式**：把「领域 + 和谐 + 保真 + 不发散」拆成**模型可逐条执行的清单**，比长段自然语言更稳；`[CONTEXT]` 解决"脱离游戏本身"，`[AVOID]` 解决"虚构画面/发散"，二者是用户诉求里最难靠零散措辞兜住的两点。

### D4. 保真与平台坑建模

- **input_fidelity**：gpt-image-2 自动高保真，harness **不传** `input_fidelity`（传了会被拒）。在适配器注释与 spec 记录此事实，作为「不发散」的天然保障；不需要也不应模拟该参数。
- **透明底不支持**：gpt-image-2 不支持 `background: transparent`。当目标尺寸 `Note` 含「透明底」语义时，harness **不原样注入**「透明背景」诱发模型瞎画，而是改写为可执行表述（如「纯净中性/单色背景，主体边缘干净便于后期抠图」），并在产物记录该尺寸的原始约束未由模型直接满足。其他可满足约束（无文案、仅 logo、圆角、安全区）照常注入。
- **>2K 实验性**：解析器默认把总像素贴近但不超 2K（3686400px）；仅当目标本身要求更高（如 4K 渠道尺寸）才进入实验档，并在 trace 日志标注 experimental。

### D5. 收敛与新解析的协同

新解析器使所有目标在生成阶段即**贴合目标比例**（`r≤3` 同比例；`r>3` 夹 3:1）。因生成像素预算恒在 ~2K、而 57% 目录档位 <655360px，**收敛主路径是「等比下采样到精确档位」**（`contain`，无可见形变，且小图更锐利）；极端比例（夹 3:1 出图）走 `cover` 裁到精确档位；iOS >2K 档可能上采样。**收敛逻辑本身不改**——`convergeMode` 的 log-ratio 分档判定天然覆盖以上情形：同比例→`contain`、夹比例后差异大→`cover`。新增的是"生成尺寸更贴合 + 总是收敛到目录精确像素"这一前置保证。

### D6. harness 决策可观测（trace 日志）

新 harness 行为（尺寸映射、参考图角色、透明底改写）此前对用户/排查完全不可见——日志只有 `prompt_len` 与 `refs` 计数。新增 `gen.harness` 结构化事件在 `BuildPrompt` 后、provider 调用前发出，携带：`kind`、`ref_count`、`multi_image_anchor`（≥2）、`provider`（openai/gemini/dashscope）、`supports_transparency`、`transparency_rewritten`；适配任务额外带 `target_size` → `gen_size`（gpt-image-2 解析结果）映射。沿用既有 `applog.From(ctx)` 双层 trace（`trace_id`/`session_id` 自动随异步 goroutine 携带），可凭 `trace_id` 复盘单次生图的 harness 取值。>2K 档位另有 `gen.adapt_above_2k` 事件（D4）。

## Risks / Trade-offs

- **R1：16 的倍数 round 引入 ±8px 误差** → 收敛步骤本就把产物精确缩放到目录档位像素，误差被吸收，用户无感。
- **R2：贴合比例的非标准尺寸触发供应商边缘 bug** → 解析器始终输出满足全部约束的合法值；失败仍有主/备 failover 与既有错误处理兜底。
- **R3：16 张参考图推高输入 token / 延迟 / 成本** → 文档已知（自动高保真处理每张图）；本期接受，仅小团队内部用；如需可后续加 quality=low 草稿档（Non-Goal）。
- **R4：四段式模板变长，可能与用户长 slot 叠加超模型偏好** → slot 已有 `maxSlotLen=500` 截断；固定段落简短且结构化，净收益为正。
- **R5：透明底改写改变用户预期** → 受影响仅 gpt-image-2 的 2 个档位；若部署主力为支持透明底的供应商（Gemini），按适配器能力探测，不一刀切。
- **R6：恒按 ~2K 出图再下采样到小档位，成本高于直接出小图** → 但 gpt-image-2 本就不接受 <655360px 出图，且下采样显著提升小图清晰度，是质量/成本的合理取舍；仅小团队内部用。

## Migration

- 纯增强，无数据迁移。`legalGenSizes`/`nearestGenSize` 被 gpt-image-2 解析器取代或内化；DashScope 映射原样保留。
- 旧 spec 中「gpt-image-2 合法枚举为三档」表述由本 change 的 MODIFIED 需求覆盖。
- 既有产物不受影响；新逻辑仅作用于新生成请求。
- 产物最终尺寸语义不变：**始终精确等于目录档位**（用户硬约束），与现状一致。

## Open Questions

- **iOS 2732×2048（>2K 实验档，唯一 2 个超上限档位）**：默认夹 2K 出图后由收敛上采样，还是放行实验档直接出更高分辨率？倾向**默认夹 2K**（实验档质量不稳、官方未保证），上采样到 2732×2048 的清晰度损失在该用途（应用商店截图）可接受；如实测不够锐利，再针对这两个 id 放行实验档。实现时集中在解析器一处的像素预算判定，便于后续调。
