## Context
本 change 在已有宣发链路（分析 → 生图 → 适配 → 生视频 → 打包）上补四块投放素材能力。约束：仅小团队内部使用、密钥硬编码、模型服务端固定、用户不可选；意图走白名单 + 确定性预分类；所有注入到模型的文本需防注入。四个能力有依赖顺序：文案 → 叠加（叠加要引用文案）→ 变体（独立但复用生图管线）→ 合规（审查前三者产物）。

## Goals / Non-Goals
- Goals:
  - 让 Agent 能主动产出结构化宣发文案，并被下游工具消费
  - 提供**确定性**文字/LOGO 叠加，摆脱生图模型画字的不可控
  - 一次产出多变体 creative，服务买量 A·B 测试
  - 投放前给出合规风险报告，下载/发布前告警（不硬拦截）
- Non-Goals:
  - 不对接真实平台投放/上传 API
  - 不做自动翻译出海本地化（后续独立 change）
  - 合规不做强制拦截、不替代法务/平台终审
  - 不引入用户可选模型

## Decisions
- **Decision: 一个 change 含 4 个新 capability + 1 个 MODIFIED**，而非拆 4 个 change。理由：四者构成单一投放闭环、共享意图白名单与长任务/事件基建，拆开会重复改 orchestration 与 transport。
- **Decision: 文案/合规复用会话或视觉链路的 LLM，不新增用户可选模型**。文案生成可挂在视觉分析同源（已能消费素材），合规以服务端固定规则 + LLM 辅助判定。符合「模型服务端硬编码」约束。
- **Decision: 文字叠加走服务端确定性渲染（如 Go 图像库 + 内置字体），不经生图模型**。理由：CTA/角标/LOGO 要求文字精确、位置可控、可复现，生图模型画字不满足。
- **Decision: 批量变体复用现有生图管线与长任务/SSE 机制**，仅新增「批次标识 + 变体维度」编排层。理由：避免重写质量门控与防注入，且前端占位/回填基建已具备。
- **Decision: 合规审查不硬拦截**，下载/发布前以风险条告警 + 确认知悉。理由：内部工具，保留用户决策权；硬拦截会阻断正常工作流。
- Alternatives considered:
  - 文字用生图模型画 → 否，错字/糊/不可控，正是要解决的痛点。
  - 合规强制拦截 → 否，与内部工具定位和用户自主权冲突。

## Risks / Trade-offs
- **字体覆盖/版权**：内置字体需覆盖中英文常用字形与 emoji，且授权允许商用渲染 → 选用开源可商用字体并做缺字回退（见 text-overlay spec）。
- **批量变体放大成本与限流**：N 个并发生图可能触发供应商限流 → 上限 8 + 串并行可控 + 单变体失败不连坐。
- **合规规则维护**：敏感词/平台政策会变 → 规则集中在服务端配置，便于更新；LLM 辅助判定降低硬规则漏报。
- **文案虚构卖点**：LLM 可能编造素材不存在的卖点 → 防注入 + 优先消费分析报告 + 「不虚构」固定指令约束。

## Migration Plan
纯增量，无破坏性变更。新增工具默认随 Agent 注册即生效；前端新增组件不影响既有渲染。合规门控以「软告警」接入下载/发布，旧流程在无高风险时行为不变。回滚：从白名单摘除四个工具 + 隐藏前端入口即可。
